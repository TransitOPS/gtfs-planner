# The Rider Experience: In-Station Wayfinding with GTFS Pathways

When a transit agency publishes GTFS Pathways data (`pathways.txt`), OpenTripPlanner (OTP) can give riders step-by-step navigation through the inside of stations — from entrance to platform. This document describes exactly what OTP returns to users and how to interpret each piece of the response.

---

## The Big Picture

A trip plan that passes through a station with pathway data looks like this:

1. A **transit leg** (e.g., bus) drops you at a station.
2. A **walk leg** guides you through the station interior, broken into individual **steps**.
3. A **transit leg** (e.g., train) picks you up on a platform.

The walk leg is where all the wayfinding detail lives. Each step corresponds to a physical movement — walking down a corridor, climbing stairs, riding an elevator, passing through a fare gate. The entire walk leg has an overall duration and distance, and each step within it has its own distance.

---

## What a Walk Step Contains

Every step in a walk leg includes these fields:

| Field | Type | Description |
|---|---|---|
| `streetName` | String | The name of the pathway, sign text, or a default like `"pathway"` |
| `distance` | Float | Distance of this step in meters |
| `relativeDirection` | Enum | The action to take: `FOLLOW_SIGNS`, `ELEVATOR`, `ENTER_STATION`, `EXIT_STATION`, `CONTINUE`, `LEFT`, `RIGHT`, etc. |
| `absoluteDirection` | Enum | Compass direction: `NORTH`, `NORTHEAST`, etc. Null for elevators |
| `lat` / `lon` | Float | Coordinates of the step's starting point |
| `feature` | Object | Additional metadata for station features (see below) |
| `area` | Boolean | True if the step crosses an open area like a plaza or platform |
| `bogusName` | Boolean | True if the street name was generated by the system rather than from data |
| `stayOn` | Boolean | Whether the path changes direction but stays on the same named route |
| `elevationProfile` | Array | Elevation data along the step (when available) |
| `alerts` | Array | Any service alerts applicable to this step |

### The `feature` Field

The `feature` field is a union type that appears on steps involving station infrastructure. It can be one of four types:

**`ElevatorUse`** — Riding an elevator between floors:
- `from: Level` — The floor you board on (`level`: numeric index, `name`: display name like "Ground" or "P1")
- `to: Level` — The floor you exit on
- `verticalDirection` — `UP`, `DOWN`, or `UNKNOWN`

**`EscalatorUse`** — Riding an escalator:
- `from: Level` — Starting floor
- `to: Level` — Ending floor
- `verticalDirection` — `UP`, `DOWN`, or `UNKNOWN`

**`StairsUse`** — Climbing or descending stairs:
- `from: Level` — Starting floor
- `to: Level` — Ending floor
- `verticalDirection` — `UP`, `DOWN`, or `UNKNOWN`

**`Entrance`** — Entering or exiting a station:
- `entranceId: String` — ID in the format `FeedId:EntranceId`
- `name: String` — Name of the entrance/exit
- `publicCode: String` — Short identifier like "A" or "B"
- `wheelchairAccessible` — Wheelchair accessibility status

---

## Signage

When a GTFS pathway has a `signposted_as` value, OTP surfaces it directly. The step will have:

- `relativeDirection: "FOLLOW_SIGNS"`
- `streetName`: the sign text (e.g., `"Follow signs to Platform 7"`)

This is the primary way riders see physical signage in their directions. If the rider traverses a pathway in the reverse direction, the `reversed_signposted_as` text is used instead.

Pathways without signage text show a default `streetName` of `"pathway"` and will have a directional `relativeDirection` like `CONTINUE`, `LEFT`, or `RIGHT`.

---

## Distance

Each step includes a `distance` field in meters. This comes from:

1. The pathway's `length` field in the GTFS data (if provided and > 0)
2. A calculated straight-line distance between the two endpoints (if `length` is not provided)

Elevator steps typically show `distance: 0.0` because the movement is vertical, not horizontal.

The walk leg itself also has a `distance` field that sums the distances of all its steps. Note: when a pathway has an explicit `traversal_time`, its distance does not count toward the walk distance calculation — the time is used directly.

---

## Traversal Time

Individual steps do not have their own time field. Time is reflected at the **leg level**:

- `duration`: Total seconds for the entire walk through the station
- `start` / `end`: Timestamps for when the walk begins and ends

The leg duration is calculated from the sum of each pathway's traversal cost, which OTP determines using this priority:

1. **`traversal_time`** from the GTFS data (if > 0) — used directly as seconds
2. **`length` / walking speed** — if no traversal time, calculated as `length / 1.4 m/s` (default walking speed)
3. **`stair_count` estimation** — if neither time nor length, each stair step is estimated as 0.4 meters of walking
4. **Minimum cost** — elevators without data get a minimal cost (the real cost comes from elevator-specific edges)

## Walkthrough: Navigating a Complex Station

Here is what OTP returns for a rider who enters a station at ground level and needs to reach an elevated platform. The scenario: walk across the main hall, go up stairs, pass through a fare gate, take an elevator to the platform level, and follow signs to the boarding area.

### GraphQL Query

```graphql
{
  plan(
    from: { lat: 40.7128, lon: -74.0060 }
    to: { lat: 40.7580, lon: -73.9855 }
    transportModes: [{ mode: WALK }, { mode: TRANSIT }]
  ) {
    itineraries {
      duration
      walkDistance
      walkTime
      legs {
        mode
        duration
        distance
        from { name }
        to { name }
        start { scheduledTime }
        end { scheduledTime }
        steps {
          streetName
          distance
          relativeDirection
          absoluteDirection
          lat
          lon
          area
          bogusName
          feature {
            ... on Entrance {
              entranceId
              name
              publicCode
              wheelchairAccessible
            }
            ... on StairsUse {
              from { level name }
              to { level name }
              verticalDirection
            }
            ... on ElevatorUse {
              from { level name }
              to { level name }
              verticalDirection
            }
            ... on EscalatorUse {
              from { level name }
              to { level name }
              verticalDirection
            }
          }
        }
      }
    }
  }
}
```

### Reading the Response Step by Step

**Step 1 — Enter the station:**
`relativeDirection: "ENTER_STATION"` with an `Entrance` feature. The rider is told to enter through "Main Entrance" (public code "A"). The entrance is wheelchair accessible.

**Step 2 — Follow signs through the main hall:**
`relativeDirection: "FOLLOW_SIGNS"` with `streetName: "Follow signs to Platforms 10-15"`. This is the `signposted_as` text from the GTFS data. The rider walks 150 meters heading northeast.

**Step 3 — Go up stairs:**
`relativeDirection: "CONTINUE"` with a `StairsUse` feature. The rider goes from the Ground level to the Mezzanine level (`verticalDirection: "UP"`). The 12-meter distance reflects the horizontal component of the stair path.

**Step 4 — Pass through fare gate:**
`relativeDirection: "FOLLOW_SIGNS"` with `streetName: "Tap card at fare gate"`. Fare gates are modeled as signposted pathways. The 2-meter distance is the physical length of the gate passage.

**Step 5 — Take elevator down:**
`relativeDirection: "ELEVATOR"` with an `ElevatorUse` feature. The rider goes from the Mezzanine (level 1.0) down to the Platform Level (level -1.0). Distance is 0.0 because the movement is vertical. Note that `absoluteDirection` is null — compass direction doesn't apply inside an elevator.

**Step 6 — Follow signs to the platform:**
`relativeDirection: "FOLLOW_SIGNS"` with `streetName: "Follow signs to Platform 12"`. Another signposted pathway guiding the rider 85 meters east to their platform.

**Step 7 — Cross the platform:**
`relativeDirection: "CONTINUE"` with `area: true`. The `area` flag tells the application this is an open space (a platform), so directions might say "cross" rather than "walk along."

## What OTP Does and Does Not Provide

### Provided

- **Step-by-step turn directions** through the station with distances per step
- **Sign text** from `signposted_as` / `reversed_signposted_as` as step names
- **Vertical transportation details** — elevator, escalator, and stairs with from/to levels, floor names, and direction (up/down)
- **Entrance/exit identification** with name, public code, and wheelchair accessibility
- **Compass direction** for each step (except elevators)
- **Coordinates** (lat/lon) for each step's starting point
- **Open area indicators** for platforms and plazas
- **Overall leg timing** — total duration, start time, and end time for the walk through the station
- **Overall leg distance** — total walking distance in meters
- **Wheelchair-aware routing** — stairs and escalators are avoided when wheelchair mode is enabled; slope penalties are applied
- **Alerts** applicable to each step
- **Leg geometry** — a polyline representing the full path through the station

### Not Provided

- **Per-step traversal time** — only the overall leg duration is given, not how long each individual step takes
- **Explicit pathway type labels** — there is no `pathwayMode` field in the response (stairs are identified by the `StairsUse` feature, elevators by `ElevatorUse`, etc.)
- **Stair count** — the number of stairs is not exposed in the API
- **Slope/grade values** — `max_slope` affects routing cost but is not shown to users
- **Pathway width** — `min_width` from the GTFS spec is not used by OTP
- **Whether a pathway is bidirectional** — this affects which routes are possible but is not shown in the response
- **Indoor maps or floor plans** — OTP provides turn-by-turn text directions and a path geometry, not a visual map of the station interior

---

## How Wheelchair Routing Changes the Experience

When a trip plan is requested with wheelchair accessibility enabled, OTP adjusts its behavior:

- **Stairs are avoided.** Stair pathways receive a high reluctance penalty (25x), making OTP strongly prefer alternative routes.
- **Escalators are avoided.** Escalator edges cannot be traversed in wheelchair mode.
- **Elevators are preferred.** Elevator pathways have no accessibility penalty.
- **Slopes matter.** Pathways with `max_slope` values above the configured threshold receive additional cost penalties. Steeper ramps are penalized more heavily.
- **Inaccessible pathways are penalized.** Pathways explicitly marked as not wheelchair accessible receive an additional reluctance penalty.

The resulting trip plan may show a different route through the station — for example, bypassing stairs entirely and routing through an elevator and ramp instead. The steps in the response will reflect this alternative path.

## Summary of Key Indicators in the Response

| What you see | What it means |
|---|---|
| `relativeDirection: "ENTER_STATION"` | Entering a station through an entrance |
| `relativeDirection: "EXIT_STATION"` | Leaving a station through an exit |
| `relativeDirection: "FOLLOW_SIGNS"` | Following physical signage; read the `streetName` for the sign text |
| `relativeDirection: "ELEVATOR"` | Riding an elevator; check `feature` for floor details |
| `relativeDirection: "CONTINUE"` | Walking straight; may have a `StairsUse` or `EscalatorUse` feature |
| `streetName: "pathway"` | A pathway segment without `signposted_as` data |
| `feature` with `StairsUse` | Stairs with from/to levels and vertical direction |
| `feature` with `EscalatorUse` | Escalator with from/to levels and vertical direction |
| `feature` with `ElevatorUse` | Elevator with from/to levels and vertical direction |
| `feature` with `Entrance` | Station entrance/exit with ID, name, code, and accessibility |
| `area: true` | Open space like a platform or plaza |
| `absoluteDirection: null` | No compass direction (typical for elevators) |
| `distance: 0.0` | Vertical movement only (elevators) |
